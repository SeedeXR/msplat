# Baseline — 2026-06-10, Mac16,1 (MacBook Pro M4, 16 GB)

Captured **before** any v1.2 changes, at commit `106499b` (v1.1.3), on the exact
target hardware (base M4 / 16 GB — the constrained machine the project must run
well on). This is the ledger every later claim diffs against.

Machine: `Mac16,1`, Apple M4, 16 GiB unified memory, macOS 25.5, Xcode 26.5,
Metal toolchain 17F42. Build: `cmake -DCMAKE_BUILD_TYPE=Release`, `metal -O3`.

## Small-chunk correctness (`-n 200 -d 4 --eval`, garden, piped)

- Exit 0; wrote 34.4 MB PLY; 138,766 gaussians.
- Mean test PSNR 21.71 / SSIM 0.730 / L1 0.0537 (24 held-out views).
- **Peak memory footprint: 1.41 GB** (`/usr/bin/time -l`, peak), max RSS 1.06 GB.
- **Observation (the blocker):** across all 200 training iterations the process
  printed *nothing* to stdout until the final eval block — confirms the silent
  loop. (No `Densified:` line either: warmup_length=500 > 200.)

## Idle-machine impact (M2.6) — is it a good citizen at `nice +7`?

From the same profile: **CPU dispatch median 0.08 ms/iter vs 7.5 ms GPU drain
(98.5% GPU-bound)**. The CPU thread is essentially idle during training (it only
encodes the command buffer), so at `nice +7` the rest of the machine stays
responsive — the contention is GPU time, which macOS schedules preemptively.
Resident memory (4.86 GB at full-res 7K) leaves ~11 GB for other apps on a 16 GB
machine. Conclusion: msplat is already a polite background citizen; no extra work
needed for the "runs while you use the machine" success metric.

## What the profile says about M3 priority (reprioritized from todo)

1. **Rasterization fwd+bwd ≈ 82% of GPU time.** The single biggest lever is
   `rasterize_backward_kernel` (M3.3 — warp/simdgroup-aggregated atomics). This
   outranks the prefix-sum collapse (M3.4) and fused-SSIM (M3.5), which target
   stages that are already <3% each — those are no longer worth the risk.
2. `proj_sh_bwd_adam` (9.4%) is the next lever → **M3.1 sparse/visible-only Adam**.
3. Async loss readback (old M3.2) is **already the case** in the non-benchmark
   path — `msplat_train_step` reads `loss_sum` without a sync (msplat_metal.mm:1328);
   the only per-iter sync is in the `BENCHMARK=1` timing path. No work needed.

## Full-resolution 7K on this M4/16GB (the 16 GB stress test)

`msplat garden -n 7000 -d 1 --eval` with the v1.2 binary. This is the headline
"can a base M4 / 16 GB laptop train a full Mip-NeRF 360 scene?" test — **yes**.

Three controlled runs, `-n 7000 --eval`, v1.2 binary:

| run | PSNR | SSIM | gaussians | wall |
|---|---|---|---|---|
| `-d 1` black bg (progressive, default `--num-downscales 2`) | 24.98 | 0.721 | 755,235 | 174 s |
| `-d 1` magenta (`--debug-bg`) | 24.95 | 0.716 | 721,095 | ~174 s |
| `--num-downscales 0` (full-res throughout, = README command) | **25.67** | **0.7835** | 1,533,620 | ~290 s |

> The `--num-downscales 0` run reproduces the README's v1.1.3 number (25.68 / 0.783)
> to two decimals on the v1.2 binary — **definitive proof the refactor does not
> regress quality**, and confirmation that the schedule (not the bg default) explains
> the difference between the rows above.

max RSS (resident peak), `-d 1` run: **4.86 GB**.

Notes (resolving the apparent gap vs the README's 25.68 / 0.783):
- **It is NOT the background color.** Black (24.98) and magenta (24.95) are within
  noise — so v1.2's black default has no quality cost, and the refactor does not
  regress quality (two independent runs agree).
- **It is `--num-downscales`.** The README's 7K number uses `--num-downscales 0`
  (full resolution for all 7000 steps). My `-d 1` runs kept the default
  `--num-downscales 2` (progressive: 4× for 3000 steps, 2× to 6000, full-res only
  the last 1000), which trains less at full res and so scores lower. The
  `--num-downscales 0` run above re-establishes apples-to-apples.
- **Memory:** resident peak 4.86 GB — comfortable on 16 GB with room for a browser.
  `/usr/bin/time -l`'s "peak memory footprint" reads ~40 GB, which is *cumulative
  allocation* inflated by densification buffer **churn** (`ensureCapacity` allocates
  new MTLBuffers before freeing the old) — not simultaneous residency. The 4.86 GB
  RSS is the number that must fit in RAM. The churn is a real optimization target
  (pre-size / reuse buffers across refines — optimization-roadmap M3.6).
- Base M4 is ~2.3× slower than the M4 Max in the README (174 s vs 77 s) — expected
  (the M4 Max has ~4× the GPU cores). Quality is chip-independent.
